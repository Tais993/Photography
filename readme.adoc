= Picture Project Maintenance Tool

A local tool for managing folder-based photography projects.

The Picture Project Maintenance Tool helps with repetitive photography workflows.
It recognises project folders, searches images, selects files and copies them to the correct folder.

This project was created for my own photography workflow, where projects often contain hundreds of images stored in a fixed folder structure.

== Status

This project is under active development.

The current version focuses on the core workflow:

* Recognising and storing photography projects
* Searching images inside projects
* Selecting images
* Copying images to editing folders
* Viewing projects and images through a website

Some planned features, such as full archive/transfer workflows, are still being developed.

== Features

* CLI for fast command-based workflows
* Website for visual project browsing
* Database storage for projects, images, metadata and selections
* Persistent image selection sessions
* Cached thumbnail generation
* RAW/JPG handling support

== Photography project structure

The application expects photography projects to use a folder-based structure.

A normal project folder should use the following naming convention:

[source,text]
----
YYYY-MM-DD-name
----

Example:

[source,text]
----
2026-04-04-Kalkar
----

A project usually contains these folders:

[source,text]
----
2026-04-04-Kalkar/
    originals/
    editing/
    finals/
----

The `originals` folder contains the original image files.
The `editing` folder is used for files that have been selected for editing.
The `finals` folder is used for finished exports.

== Subprojects

Subprojects are supported by placing a folder inside the main project folder with a name starting with a dot.

Example:

[source,text]
----
2026-04-04-Kalkar/
    originals/
    editing/
    finals/
        .saturday/
            originals/
            editing/
            finals/

----

A subproject uses the date from the parent project, but has its own name and folder structure.

== Project identification

When a project is initialised, the application creates a `project.info` file inside the project folder.

This file contains the project ID from the database.
It allows the application to recognise the project again later, even if the folder is opened directly.

Example:

[source,text]
----
2026-04-04-Kalkar/
    project.info
    originals/
    editing/
    finals/
----

== Running the project

=== Prerequisites

* .NET SDK
* PostgreSQL
* Valid configuration file
* IrfanView, if IrfanView-related features are used

=== Configuration

The application uses shared configuration for settings such as:

* Database connection
* Thumbnail cache location
* Thumbnail sizes
* JPEG quality
* External tool paths

Create or update the local configuration file before running the application.

Example configuration values:

[source,json]
----
{
  "ConnectionStrings": {
    "Default": "Host=localhost;Port=5432;Database=photography;Username=postgres;Password=password"
  },
  "Thumbnails": {
    "DefaultSize": 300,
    "LargeSize": 1200,
    "JpegQuality": 80,
    "CachePath": "thumbnail-cache"
  }
}
----

Do not commit real passwords or personal configuration values.

=== CLI

Run the CLI with:

[source,bash]
----
dotnet run --project Cli
----

The CLI is intended for fast workflows, such as initialising projects, searching images, selecting images and copying files.

=== Website

Run the website with:

[source,bash]
----
dotnet run --project Website
----

The website can be used to visually browse projects and images.

== Trying the application

To try the application, create a small test project folder first:

[source,text]
----
2026-04-04-TestProject/
    originals/
        DSC_1234.JPG
        DSC_1234.NEF
        DSC_1235.JPG
    editing/
    finals/
----

Then run the CLI or website and initialise the project.
After the project has been initialised, the application can store the project and image information in the database.

A basic test workflow is:

[source,text]
----
1. Create a test project folder.
2. Add a few JPG or RAW files to the originals folder.
3. Run the CLI and initialise the project.
4. Search for an image by filename or number.
5. Select one or more images.
6. Copy the selected images to the editing folder.
7. Open the website to view the project and generated thumbnails.
----

== Testing

Run all tests with:

[source,bash]
----
dotnet test
----

The tests mainly focus on application services, such as image handling, image selection, copying files and thumbnail caching.

== Architecture

The project uses a layered architecture:

[source,text]
----
Presentation
↓
Application / Logic
↓
Infrastructure implementations
----

Infrastructure functionality is accessed through interfaces defined by the application layer.
This keeps the core logic testable and prevents it from depending directly on PostgreSQL, the file system, ImageMagick or IrfanView.
